# MDK 多Target固件文件名配置踩坑与解决方案文档
版本：V1.0
适用项目：STM32 OTA 双分区升级工程
工程结构：BOOTLOADER、APP1、APP2 三个 Target，每个 Target 分配独立编译输出子文件夹
文档用途：记录调试全流程踩坑点、底层原理、稳定落地配置，用于后续开发避坑学习

## 一、基础核心原理（所有问题的根源）
### 1. Target名称 和 编译产物文件名完全隔离
Keil MDK 两套独立配置，无自动同步逻辑：
1. 工程切换栏 Target 名称：仅区分编译宏、芯片、链接脚本，**不控制 axf/hex/bin 文件名**；
2. `.axf / .hex / .map` 文件名称控制：魔术棒 → Output → Name of Executable；
3. `.bin` 文件：不属于 MDK 原生输出，需要 fromelf 工具转换，必须在 User 选项卡手动配置转换命令。

### 2. MDK两套互不相通的内置变量（最高频踩坑点）
| 变量标识 | 唯一生效位置 | 功能作用 | 错误使用后果 |
| ---- | ---- | ---- | ---- |
| `@L` | Output页面 Name of Executable输入框 | 自动替换为当前Target名称，生成axf、hex | 写到User脚本内会被当成纯文本字符，不做任何替换，生成`@L.bin` |
| `$L` | User选项卡 编译后执行命令框 | 脚本环境下代表当前Target名称 | 写到Output输入框无解析效果，文件名不会变化 |

### 3. 独立子文件夹路径规则
本工程目录结构：
- APP1产物路径：`MDK-ARM\APP1\`
- APP2产物路径：`MDK-ARM\APP2\`

1. axf真实完整相对路径：`.\APP1\APP1.axf`；
2. 脚本只写`APP1.axf`：程序会去上级MDK-ARM根目录查找文件，路径错位，报错文件不存在；
3. MDK内置路径宏`$P`、`#L`在多文件夹架构存在底层BUG，拼接后极易生成带双重盘符的非法路径。

## 二、完整踩坑清单（按调试顺序整理）
### 坑1：主观认为修改Target名称，产物文件名自动跟随
现象：切换APP1/APP2 Target，编译文件依旧为旧BOOTLOADER名称；
根本原因：Target名和输出文件名是两套独立配置，无联动逻辑。

### 坑2：`@L` 与 `$L` 变量交叉混用
现象：User脚本写入 `./@L.bin ./@L.axf`，编译输出`@L.bin`，不会替换为APP1；
根本原因：`@L`仅Output页面识别，命令行脚本环境不解析该变量。

### 坑3：复制命令丢失美元符号，`$L` 手误写成 `SL`
现象：脚本命令写 `./SL.bin ./SL.axf`，产物固定为`SL.bin`；
根本原因：缺少`$`符号后，SL仅为普通文本，MDK不会识别为变量。

### 坑4：多子文件夹场景下路径宏`$P` / `#L`解析异常
1. 使用`$P$L.bin`拼接：生成非法路径 `E:\xxx\E:\xxx.bin`，Windows系统拒绝创建，报错 `Q0147E`；
2. 使用`#L`.bin拼接：生成 `APP1.axf.bin`，axf后缀冗余，文件名不整洁。

### 坑5：脚本相对路径未携带子目录层级
现象：命令 `fromelf --bin -o APP1.bin APP1.axf`，报错 `Q0122E 无法打开文件`；
根本原因：axf保存在子文件夹APP1内，命令缺少`./APP1/`前缀，向上级目录查找文件。

### 坑6：旧编译残留文件干扰判断
现象：Clean清理缓存后，文件夹同时存在旧`@L.axf`与新`APP1.axf`，误以为配置未生效；
解决规范：Clean后手动删除对应目录所有`@L.*`缓存文件，再完整重编译。

## 三、最终稳定落地解决方案（无宏、零报错，适配独立子文件夹架构）
### 核心思路
放弃所有存在兼容性BUG的自动变量（`@L` / `$L` / `$P` / `#L`），每个Target固定写死对应子目录路径，无变量解析异常，适合量产工程长期使用。

### 1. APP1 Target 全套标准配置
1. 魔术棒 → Output
`Name of Executable` 输入固定名称：`APP1`
2. 魔术棒 → User → After Build/Rebuild → Run#1
转换命令（直接复制使用）：
```shell
fromelf --bin -o ./APP1/APP1.bin ./APP1/APP1.axf
